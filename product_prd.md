PRD: Google-in-a-Day (Lightweight Real-time Search Engine)1. Product Vision & DefinitionProduct Name: Google-in-a-Day: Lightweight Real-time Search EngineDefinition: A high-performance, minimalist search engine architecture designed to recursively crawl the web from a seed URL, index data in real-time, and provide instantaneous search capabilities over the live-growing index.2. Goals & Success MetricsThe success of this project is measured not just by functionality, but by system resilience under high I/O stress and efficient resource management.2.1 Strategic GoalsArchitectural Resilience: Maintain stability under heavy I/O loads by implementing robust Back-pressure mechanisms.Concurrency Management: Execute parallel Read (Searcher) and Write (Indexer) operations without data corruption or Race Conditions.Human-in-the-loop (HITL): Provide a dashboard for operators to monitor the crawling progress and manually manage thresholds.2.2 Success MetricsThroughput: Number of pages crawled and indexed per second.Search Latency: Results returned in <100ms, regardless of index growth.Memory Efficiency: Optimized memory footprint for the "Visited Set" and "Inverted Index" structures.3. Technical Requirements: The Indexer (Crawler)The Indexer is the engine that discovers the web's graph structure and processes raw data.Recursive Crawling: Must perform Depth-First (DFS) or Breadth-First (BFS) traversal up to a defined depth $k$.Uniqueness Control: Implement a thread-safe Visited Set to prevent redundant crawling of the same URLs.Native-Only Constraint: High-level frameworks (Scrapy, Beautiful Soup, etc.) are strictly prohibited. Only native standard libraries (e.g., urllib for Python, net/http for Go) shall be used.Back-pressure & Throttling:Max Rate of Work: Limit requests per second to avoid spamming target servers.Queue Depth Management: Define an upper bound for the URL buffer. When the queue is full, the crawler must trigger a Throttling state.4. Technical Requirements: The Searcher (Query Engine)The Searcher serves the live data generated by the Indexer to the end-user.Live Indexing Support: Concurrent search capability while the index is being updated.Thread-Safety: Synchronization of the Inverted Index using Mutexes (RWMutex) or Channels to ensure safe shared-memory access.Relevancy Heuristic: Rank results based on a simple scoring logic:TF (Term Frequency): Frequency of the keyword within the document.Heading Weight: Bonus points if the keyword appears in <title> or <h1> tags.Query Interface: Instant processing of keywords to return the top 10 most relevant results.5. System Visibility & UI (Dashboard)To ensure the system is not a "black box," a real-time monitoring interface (CLI or Web UI) is mandatory.Indexing Progress: Total processed URLs vs. Pending URLs in the queue.Queue Depth: Active memory usage and buffer saturation percentage.Throttling Status: Real-time indicator of the Back-pressure state.Error Analytics: Monitoring HTTP errors (404, 500) and timeouts.6. Persistence & Resilience (Bonus)Mechanism for recovery after system interruptions.State Checkpointing: Periodic serialization of the "Visited Set" and "Current Index" to disk (e.g., every 100 pages).Resume Capability: Ability to restart the system and continue from the last depth $k$ and un-crawled queue elements.7. Acceptance Criteria (Testing Scenarios)Concurrency Test: Verify no deadlocks occur while running with at least 50 concurrent workers.Depth Test: Confirm the system successfully retrieves 3rd-level links when $k=3$ is specified.Memory Leak Test: Observe stable RAM usage during long-term (1 hour+) crawling sessions.
