# PRD: Google-in-a-Day (Lightweight Real-time Search Engine)

## 1. Product Vision & Definition
**Product Name:** Google-in-a-Day: Lightweight Real-time Search Engine  
**Definition:** A high-performance, minimalist search engine architecture designed to recursively crawl the web from a specified origin, index data in real-time, and provide instantaneous search capabilities over the live-growing index using only language-native functionalities.

---

## 2. Goals & Success Metrics
The success of this project is measured by system resilience under high I/O stress, efficient resource management, and strict adherence to the "native-only" constraint.

### 2.1 Strategic Goals
* **Architectural Resilience:** Maintain stability under heavy I/O loads by implementing robust Back-pressure mechanisms to manage queue limits.
* **Concurrency Management:** Execute parallel Read (Searcher) and Write (Indexer) operations without data corruption or Race Conditions.
* **Human-in-the-loop (HITL):** Provide a CLI dashboard for operators to monitor crawling progress, system load, and back-pressure status.

### 2.2 Success Metrics
* **Throughput:** Stable crawling and indexing rate without exhausting system RAM.
* **Search Latency:** Results returned in under 100ms, regardless of how large the active index grows.
* **Memory Efficiency:** Optimized memory footprint for the "Visited Set" and the "Inverted Index" dictionary.

---

## 3. Technical Requirements: The Indexer (Crawler)
The Indexer is the engine that discovers the web's graph structure and processes raw data.

* **Core API:** Must expose an `index(origin, k)` capability.
* **Recursive Crawling:** Initiates a web crawl from the `origin` URL up to a defined depth $k$. It must discover new links and process them concurrently.
* **Uniqueness Control:** Implement a thread-safe "Visited Set" to ensure the system never crawls the same page twice.
* **Native-Only Constraint:** High-level frameworks (Scrapy, Beautiful Soup, Requests) are strictly prohibited. Only native standard libraries (e.g., `urllib`, `html.parser`) shall be used.
* **Back-pressure & Throttling:**
  * **Queue Depth Management:** Define a strict upper bound for the URL buffer (`queue.Queue(maxsize)`).
  * **Load Control:** When the queue reaches its limit, the crawler must block new additions (triggering a Throttling state) to manage load in a controlled way and prevent memory leaks.

---

## 4. Technical Requirements: The Searcher (Query Engine)
The Searcher serves the live data generated by the Indexer to the end-user.

* **Core API:** Must expose a `search(query)` capability.
* **Output Format:** Given a string `query`, it must return a list of triples taking the exact form: `(relevant_url, origin_url, depth)`.
* **Live Indexing Support:** Designed such that `search` can be invoked while the `indexer` is still active, reflecting new results as they are discovered.
* **Thread-Safety:** Synchronization of the index data structure using Mutexes (`threading.Lock`) to ensure safe shared-memory access between background crawling threads and the foreground search interface.
* **Relevancy Heuristic:** Rank results based on a Term Frequency (TF) scoring logic (frequency of the keyword within the indexed document).

---

## 5. System Visibility & UI (CLI Dashboard)
To ensure the system is not a "black box," a real-time monitoring Command Line Interface (CLI) is mandatory. It must make it easy to initiate operations and view the state of the system.

* **Indexing Progress:** Total processed URLs vs. Total indexed pages.
* **Queue Depth:** Current active URLs waiting in the buffer versus total capacity.
* **Back-pressure Status:** Real-time indicator (Normal vs. ACTIVE) showing if the system is currently managing load via throttling.

---

## 6. Persistence & Resilience (Bonus Feature)
Mechanism for system recovery to ensure hours of crawling are not lost upon interruption.

* **State Checkpointing:** Upon a graceful exit, serialization of the current "Visited Set", "URL Queue", and "Indexed Data" to a local snapshot file (`snapshot.json`).
* **Resume Capability:** Ability to restart the system, load the previous state, and continue crawling from the exact $k$ depth and un-crawled queue elements without starting from scratch.

---

## 7. Acceptance Criteria (Testing Scenarios)
* **Concurrency Test:** Verify no deadlocks or race conditions occur while running with multiple worker threads during Live Search operations.
* **Depth Test:** Confirm the system successfully retrieves valid links up to the specified $k$ depth boundaries.
* **Resilience Test:** Observe that system RAM usage does not spiral out of control and the queue depth hard-limit successfully halts unchecked crawling.
